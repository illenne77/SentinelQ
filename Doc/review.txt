문서 수준이 상당히 높습니다.
특히 다음 4가지는 일반적인 개인 프로젝트 계획서를 넘어서는 수준입니다.

* 개발 방법론(Harness Engineering)과 제품 런타임을 명확히 분리
* 결정론적 엔진과 LLM 역할 분리
* 감사 로그·검증·재현성 중심 설계
* 의료 SW 스타일의 운영 안정성 철학 적용

단순 “AI 주식 추천기”가 아니라, 실제 운영 가능한 반자동 트레이딩 플랫폼 관점으로 설계되어 있습니다.
특히 사용자가 의료 SW 개발 경험이 있다는 점이 문서 전체 구조에 강하게 반영되어 있습니다.

다만 현재 문서는 “아키텍처 철학”은 매우 강하지만, 실제 구현 단계에서 문제가 될 수 있는 영역들이 일부 빠져 있습니다. 아래 기준으로 검토하겠습니다.

# 강점 분석

## 1. “LLM은 분석만, 실행은 결정론” 방향이 매우 적절함

이 부분이 핵심입니다.

많은 AI 투자 시스템이 실패하는 이유:

* LLM에게 실시간 매매 판단까지 맡김
* 설명 가능한 구조가 없음
* 재현 불가능
* 백테스트 불가능
* 동일 상황에서 결과가 바뀜

그런데 이 문서는:

* 매수 → 인간 승인
* 매도 → 룰 엔진
* 리스크 → 결정론
* 주문 → 저지연 엔진

으로 분리했습니다.

이건 실제 운영 가능한 구조입니다.
특히 아래 문장이 중요합니다.

> 주문 실행 경로는 절대 LLM을 거치지 않음

이 방향은 맞습니다.

---

## 2. Harness Engineering 적용이 실제로 효과 있을 가능성이 높음

대부분 “AI로 개발한다” 수준에서 끝나는데, 이 문서는:

* 구조 테스트
* 계층 의존성 강제
* PR 크기 제한
* 룰 기반 CI
* 도메인 검증
* replay
* audit

까지 설계했습니다.

특히 이것:

```python
ALLOWED_DEPS = {
```

이런 구조 테스트는 실제로 효과 있습니다.

AI 코딩 에이전트는 시간이 지나면 거의 반드시:

* service → repository bypass
* util god object 생성
* shared helper 남발
* circular dependency

를 일으킵니다.

문서가 이 문제를 정확히 이해하고 있습니다.

---

## 3. 기술 스택 선택이 현실적임

이 부분도 좋습니다.

### 흔한 실패 패턴

* 모든 걸 Python으로 하려 함
* 모든 걸 Rust로 하려 함
* 마이크로서비스 과분해
* Kubernetes부터 도입

현재 문서는 비교적 균형이 좋습니다.

| 영역  | 선택       |
| --- | -------- |
| AI  | Python   |
| 백엔드 | C#       |
| UI  | WPF      |
| 저지연 | C++/Rust |

특히 당신 같은 C#/Windows 기반 개발자에게는 매우 현실적입니다.

---

## 4. 의료 SW 사고방식이 금융 시스템과 잘 맞음

이 부분이 의외로 큰 강점입니다.

금융 시스템과 의료 SW는 공통점이 많습니다.

| 의료 SW                  | 금융     |
| ---------------------- | ------ |
| audit trail            | 거래 로그  |
| reproducibility        | 백테스트   |
| deterministic behavior | 주문 일관성 |
| validation             | 리스크 검증 |
| fail-safe              | 손실 제한  |

그래서 문서의 안정성 철학이 상당히 설득력 있습니다.

# 2. LLM 비용 구조가 빠져 있음

> “AI 기반 투자 프로그램 설계”

치고는 상당히 현실적입니다.

특히:

* 의료 SW 사고방식
* 결정론 중심
* 운영 안정성 우선
* LLM 불신 기반 검증 구조

는 매우 좋은 방향입니다.

다만 현재 상태는:

> “좋은 플랫폼 설계”

에 가깝고,

> “검증된 투자 전략”

은 아직 아닙니다.

실제로 성공 여부는 앞으로 다음 3개에 달립니다.

1. 실제 알파 전략 존재 여부
2. LLM 비용 대비 효율
3. 범위 통제 능력

문서의 가장 큰 위험은 기술 실패보다:

> 범위 과대 + 전략 부재

입니다.

반대로 이 두 가지를 통제하면, 개인 개발 프로젝트 치고는 상당히 높은 완성도까지 갈 가능성이 있습니다.
